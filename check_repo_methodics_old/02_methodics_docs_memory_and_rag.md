# 02. Docs, Memory And RAG

> Методики для проверки того, как проект подаёт контекст агенту через документы, knowledge layer и retrieval.

## 1. `AGENTS.md` как оглавление проекта

- Что проверяем: `AGENTS.md` коротко объясняет проект, структуру директорий, важные документы, базовые принципы и список запретных действий.
- Чем помогает: агент получает карту до чтения кода и не тратит контекст на случайные блуждания.
- Чем опасно отсутствие: каждый новый агент начинает с нуля и строит свою версию проекта по косвенным признакам.
- Сложность внедрения: низкая.
- Как проверять: файл должен направлять к нужным артефактам, а не пытаться заменить ADR, deployment guide и runbooks одновременно.

## 2. Дополнительные AI-instruction files рядом с кодом

- Что проверяем: при необходимости у проекта есть tool-specific файлы вроде `AGENTS.md`, `.github/copilot-instructions.md`, `.cursorrules`, `.clinerules`, `.windsurfrules` или аналогов, и они образуют иерархию, а не хаос.
- Чем помогает: разные агенты читают один и тот же проект по согласованным правилам; глобальная карта не раздувается, а детальные правила лежат ближе к своей зоне применения.
- Чем опасно отсутствие: один агент работает по `AGENTS.md`, другой по отдельному rule file, а правила несовместимы или слишком общие; глобальный файл пухнет, потому что в него пытаются сложить всё.
- Сложность внедрения: низкая.
- Как проверять: искать трёхслойную схему. `AGENTS.md` — canonical source и карта проекта; path-scoped instruction files — actionable checklist для конкретной директории или типа файлов; role-specific agent files — опциональный слой для повторяющихся workflows. Отдельно проверить, что во всех слоях есть explicit scope и нет конфликтов по правилам.
- Red flags: instruction file содержит только абстрактное «пиши хороший код»; один файл требует thin handlers, второй поощряет business logic в routes; path-scoped rules переопределяют глобальные правила без явной причины.

## 3. Progressive disclosure вместо context stuffing

- Что проверяем: документация устроена слоями: `AGENTS.md` -> root `README.md` -> local `README.md` -> `docs/` -> код.
- Чем помогает: агент подгружает ровно столько контекста, сколько нужно для текущей задачи.
- Чем опасно отсутствие: большой стартовый документ перегружает контекст, а важные сигналы теряются в середине текста.
- Сложность внедрения: низкая или средняя.
- Как проверять: можно ли понять, куда идти дальше, не читая сразу весь corpus документации.

## 4. Documentation as interface

- Что проверяем: docs помогают выполнять работу, а не просто описывают проект маркетинговым языком.
- Чем помогает: документация становится operational interface для людей и агентов.
- Чем опасно отсутствие: README рассказывает, зачем нужен проект, но не помогает понять, как его менять и проверять.
- Сложность внедрения: средняя.
- Как проверять: в docs должны быть архитектурные карты, контракты, runbooks, execution flows и decision records, а не только обзор высокого уровня.

## 5. Repository as system of record

- Что проверяем: архитектурные решения, спецификации, changelog, runbooks и domain knowledge хранятся в Git как основные источники правды.
- Чем помогает: retrieval строится по стабильным и версионируемым артефактам.
- Чем опасно отсутствие: агенту приходится сверять Git, wiki, чат и Notion, где факты расходятся.
- Сложность внедрения: средняя организационно.
- Как проверять: для каждого важного знания можно назвать единственный основной файл или каталог.

## 6. Single source of truth для ключевых фактов

- Что проверяем: один и тот же факт не продублирован в пяти местах без явного владельца.
- Чем помогает: уменьшаются противоречия и stale context.
- Чем опасно отсутствие: агент получает несколько версий одного правила и выбирает наугад.
- Сложность внедрения: средняя.
- Как проверять: брать конкретный факт, например правило работы с миграциями или layout пакетов, и смотреть, где он живёт и кто его обновляет.

## 7. ADR, specs и execution flows

- Что проверяем: у проекта есть ADR или decision log, а для критических цепочек есть flow descriptions.
- Чем помогает: агент понимает не только структуру, но и причину решений и реальный путь выполнения.
- Чем опасно отсутствие: код есть, но объяснить, почему система сделана именно так и где проходит critical path, сложно.
- Сложность внедрения: средняя.
- Как проверять: искать документы вида `docs/adr/`, `docs/specs/`, `docs/flows/`, `runbooks/` или аналогичные каталоги.

## 8. Doc-gardening как регулярная гигиена

- Что проверяем: проект умеет выявлять stale docs, битые ссылки, пустые заготовки, противоречия и document drift.
- Чем помогает: knowledge layer не превращается в архив ложных подсказок.
- Чем опасно отсутствие: с ростом проекта retrieval всё чаще поднимает устаревшие документы.
- Сложность внедрения: низкая для ссылок и stale markers, средняя для более умного анализа.
- Как проверять: есть ли periodic review, doc lint, check scripts или хотя бы явный процесс document cleanup. Хороший минимальный набор: link validity, TODO/FIXME scan, поиск stub files, age-based flags для давно не обновлявшихся docs, список документов с потенциальными противоречиями.
- Red flags: активный модуль сильно менялся, а связанный runbook не трогали месяцы; в docs много `TODO`, `FIXME`, `TBD` и битых ссылок.

## 9. Что стоит хранить внутри репозитория как память

- Что проверяем: в репозитории хранятся устойчивые знания: ADR, specs, runbooks, post-mortem, changelog, eval datasets, regression fixtures.
- Чем помогает: эти знания можно версионировать, ревьюить и подмешивать в retrieval без сомнений в происхождении.
- Чем опасно отсутствие: важные решения уходят в устные договорённости и чаты.
- Сложность внедрения: низкая или средняя.
- Как проверять: определить, где живут архитектурные решения, инварианты и накопленный operational опыт.

## 10. Что не стоит хранить внутри репозитория как память

- Что проверяем: Git не используется как свалка scratchpad-файлов, транскриптов, embeddings и прочих производных артефактов.
- Чем помогает: репозиторий остаётся проверяемым и пригодным для retrieval.
- Чем опасно отсутствие: knowledge layer шумит, размер артефактов растёт, а source of truth размывается.
- Сложность внедрения: низкая.
- Как проверять: искать временные summaries, raw chat logs, vector dumps и generated search artifacts без строгой причины.

## 11. Agent-directed retrieval

- Что проверяем: агент сам выбирает, какой документ, раздел или артефакт открыть, вместо автоматической подкачки всего corpus в prompt.
- Чем помогает: working context остаётся компактным и релевантным.
- Чем опасно отсутствие: retrieval layer начинает вредить качеству за счёт шума и duplicate context.
- Сложность внедрения: средняя, если retrieval уже есть; низкая как проектный принцип.
- Как проверять: смотреть, подаются ли документы по запросу и задаче, или контекст формируется жадным автосбором всего, что кажется «полезным».

## 12. Retrieval ladder: от простого к сложному

- Что проверяем: проект не прыгает сразу в сложный RAG, а поднимается по ступеням: file tree -> grep -> local docs -> keyword search -> hybrid search -> symbol graph.
- Чем помогает: команда не маскирует плохую структуру кода дорогой инфраструктурой retrieval.
- Чем опасно отсутствие: появляется ложная уверенность, что векторный индекс заменил архитектуру и документацию.
- Сложность внедрения: низкая как методика, высокая только на поздних ступенях.
- Как проверять: понять, какой pain закрывает каждая следующая ступень retrieval, и есть ли для неё реальная необходимость.
- Практический ориентир:

| Размер кодовой базы | Минимальная ступень | Комментарий |
|---------------------|---------------------|-------------|
| `<10 KLOC` | file tree + grep | Обычно этого достаточно |
| `10–50 KLOC` | local docs + keyword search | Нужна осмысленная навигация по документам |
| `50–100 KLOC` | hybrid search | Терминология уже плавает, одной лексики часто мало |
| `>100 KLOC` | + symbol/call/dependency graph | Impact analysis становится системной задачей |

- Red flags: проект на 80 KLOC живёт только на `grep`; команда коммитит vector index, но у неё нет local README и ADR.

## 13. Контекстные антипаттерны

- Что проверяем: команда умеет распознавать и снижать типовые антипаттерны работы с контекстом.
- Чем помогает: retrieval и docs перестают мешать агенту.
- Чем опасно отсутствие: даже хорошая документация начинает работать против качества.
- Сложность внедрения: низкая как дисциплина, средняя как системная практика.
- Как проверять: искать следующие паттерны и их признаки.

| Антипаттерн | Признак при аудите |
|-------------|-------------------|
| Context Stuffing | `AGENTS.md` или root `README` пытается содержать весь проект целиком |
| Stale Context | Документы давно не обновлялись, хотя код рядом активно меняется |
| Context Leakage | В agent instructions попадают лишние секреты, внутренние детали или нерелевантные данные |
| No Context | Нет ни карты проекта, ни локальных overview, ни проектных правил |
| Duplicate Context | Одно и то же правило описано в 3+ местах с расхождениями |

## 14. Для RAG важны два вопроса: relevance и faithfulness

- Что проверяем: если внутри проекта есть RAG-контур, отдельно измеряются качество retrieval и качество ответа по retrieved context.
- Чем помогает: становится видно, ломается ли поиск или ломается синтез поверх найденных данных.
- Чем опасно отсутствие: все сбои RAG выглядят одинаково, а чинят не тот слой.
- Сложность внедрения: средняя.
- Как проверять: смотреть, есть ли метрики или evals для retrieved chunks и для faithful generation.
- Red flags: команда пишет «RAG плохой», но не может сказать, не нашёл ли поиск нужный документ или модель сгаллюцинировала поверх правильного документа.

## 15. Golden datasets и knowledge fixtures рядом с репозиторием

- Что проверяем: если проект использует AI runtime или RAG, golden datasets и regression fixtures лежат рядом с кодом и versioned вместе с ним.
- Чем помогает: изменение логики можно проверять на одном и том же корпусе, а feedback loop замыкается в Git.
- Чем опасно отсутствие: quality tuning происходит на ощущениях и случайных примерах из чатов.
- Сложность внедрения: средняя.
- Как проверять: искать каталоги evals, fixtures, datasets и связь их версий с изменениями логики или prompts-as-code.

## Базовые выводы для аудита

- если `AGENTS.md` длинный, но не навигирует, он не помогает агенту;
- если docs нельзя использовать как source of truth, retrieval становится рискованным;
- если команда пытается лечить хаос в репозитории сложным RAG, сначала нужно чинить репозиторий;
- если в Git нет устойчивых знаний, агент будет опираться на случайный session context.
