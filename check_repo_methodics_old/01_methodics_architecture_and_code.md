# 01. Architecture And Code

> Методики для проверки того, насколько архитектура и кодовая база пригодны для работы AI-агента.

## Как читать этот файл

У каждого пункта есть пять аспектов:

- что проверяем;
- чем это помогает;
- чем опасно отсутствие;
- насколько сложно внедрить как правило;
- как это обычно проверяется на практике.

## 1. Structural alignment: архитектура совпадает со структурой исходников

- Что проверяем: дерево директорий, пакетов и namespace отражает реальные домены, подсистемы и integration boundaries.
- Чем помогает: агент быстрее находит нужный слой и реже лезет в соседние модули наугад.
- Чем опасно отсутствие: схема живёт в docs, а код устроен иначе; любой change request начинается с археологии.
- Сложность внедрения: высокая в старом монолите, средняя в новых и активно поддерживаемых проектах.
- Как проверять: пройти по шагам `docs -> directory map -> imports -> tests`. Сначала выписать домены и компоненты из docs, затем сопоставить их с каталогами, затем проверить реальные импорты и архитектурные тесты. Если есть ADL, architecture tests или import rules, они становятся основным evidence.
- Red flags: один домен размазан по нескольким чужим пакетам; пакет назван по домену, но внутри содержит чужие responsibility; архитектурная диаграмма не совпадает с реальными путями.

## 2. Bounded contexts и явные границы модулей

- Что проверяем: у крупных пакетов есть понятный scope, public API и внутренности, в которые не ходят напрямик.
- Чем помогает: агент видит безопасную точку входа и меньше ломает внутренние детали чужого модуля.
- Чем опасно отсутствие: появляются cross-boundary imports, скрытые зависимости и эффект домино при локальных правках.
- Сложность внедрения: средняя; часто начинается с соглашений, затем закрепляется линтерами и architecture tests.
- Как проверять: искать импорты из `internal`, `_private`, `impl`, `adapters` и аналогичных зон, которые используются извне.

## 3. Архитектурные ограничения должны быть исполнимыми

- Что проверяем: критические ограничения закреплены не только в ADR и диаграммах, но и в тестах, линтерах, import rules или иных fitness functions.
- Чем помогает: рассинхрон между архитектурой и реализацией ловится автоматически, а не постфактум на инциденте.
- Чем опасно отсутствие: команда считает, что архитектура есть, но код постепенно уходит в обходные пути и shortcuts.
- Сложность внедрения: средняя для простых правил импорта и слоёв, высокая для метрик coupling или runtime constraints.
- Как проверять: смотреть CI, custom checks, import-linter, pytest-архитектурные тесты, Sonar и любые threshold-based проверки.

## 4. Декларативные правила, ADL и feedback framework

- Что проверяем: архитектурные правила можно выразить декларативно, хотя бы в виде псевдокода, DSL, таблицы ограничений или явной спецификации, а провал правила рассматривается как feedback opportunity.
- Чем помогает: архитектор, разработчик и агент обсуждают одно и то же правило в общей форме, а не через набор разрозненных ad hoc тестов.
- Чем опасно отсутствие: ограничения распадаются на случайные локальные проверки без общей модели, а failed check трактуется как «сломался тест», а не как сигнал о рассинхроне.
- Сложность внедрения: средняя.
- Как проверять: искать каталог `docs/adr/`, архитектурную спецификацию, pseudo-code rules, import policy matrix или другой документ, из которого можно вывести executable checks.
- Red flags: архитектурные тесты есть, но никто не может объяснить, какое правило они отражают и какой риск контролируют.

## 5. Small simple blocks вместо god files

- Что проверяем: большие файлы и крупные смешанные модули разбиты по ответственности; один файл описывает одну концепцию.
- Чем помогает: файл помещается в рабочий контекст агента, а правка становится локальной.
- Чем опасно отсутствие: lost in the middle, скрытые связи, частичные правки и рост числа поломок при seemingly small change.
- Сложность внедрения: средняя; чаще всего делается инкрементально по мере касаний кода.
- Как проверять: использовать и качественные, и количественные пороги. Для Python-проекта разумный ориентир такой: `<100` строк на файл — ideal, `100–300` — acceptable, `300–500` — strong refactor candidate, `500+` — явный red flag. Для зрелого репозитория полезно считать долю файлов `>300` строк и отслеживать её динамику.
- Red flags: `utils.py` > 200 строк; `core.py` или `service.py` > 500 строк; один файл одновременно содержит API, domain logic, persistence и integration glue.

## 6. Honest contracts и deep modules на публичных границах

- Что проверяем: ключевые функции, сервисы и use cases имеют type hints, ясные имена, docstring, ошибки, примеры или явные pre/post-conditions; при этом внешний API остаётся компактным, а сложность спрятана внутри deep module.
- Чем помогает: агент понимает контракт без чтения всей реализации и реже домысливает смысл; deep module даёт одну понятную точку входа вместо десятка мелких lifecycle-методов.
- Чем опасно отсутствие: одна и та же функция интерпретируется по-разному людьми, тестами и агентом; shallow API вынуждает агента восстанавливать порядок вызовов вручную.
- Сложность внедрения: низкая для новых модулей, средняя для старого кода.
- Как проверять: брать frequently changed public functions и смотреть, можно ли понять входы, выходы и side effects без чтения тела; отдельно смотреть, не расплескана ли одна операция по API вроде `init()`, `prepare()`, `apply()`, `finalize()`, `cleanup()` без явной необходимости.
- Red flags: `process(data) -> Any`; десяток тривиальных методов вместо одного содержательного use case API.

## 7. Zero-context survival

- Что проверяем: модуль остаётся понятным без исходного промпта, устных пояснений и знания локального фольклора команды.
- Чем помогает: новый агент или новый инженер быстрее начинает работать без скрытого контекста.
- Чем опасно отсутствие: проект работает только для тех, кто уже знает его историю.
- Сложность внедрения: средняя; обычно требует лучшего именования, явных зависимостей и удаления магии.
- Как проверять: сочетать zero-context test с greedy reading. Агент открывает файл, затем ближайшие зависимости, тесты и docs модуля. Если за 2–3 шага он понимает ответственность и границы, критерий близок к выполнению. Если для понимания одной функции нужно читать 10+ файлов, критерий не выполнен.
- Red flags: функция `process(data)` без типов и docstring; смысл переменных определяется только через историю команды или прошлый промпт.

## 8. Явные зависимости и минимум скрытой магии

- Что проверяем: business logic не зависит от неявных globals, side effects на import, monkey patching, скрытых singleton или магической регистрации.
- Чем помогает: агент видит реальные точки влияния и лучше прогнозирует последствия изменений.
- Чем опасно отсутствие: правка в одном месте ломает поведение в другом, а цепочку почти невозможно восстановить локальным чтением.
- Сложность внедрения: средняя или высокая, если проект долго жил на implicit wiring.
- Как проверять: искать конфиг, service locator, import-time initialization, метаклассы и декораторы, радикально меняющие поведение без явного контракта.

## 9. Оркестрация отделена от доменной логики, а side effects локализованы

- Что проверяем: entrypoints, роуты, CLI и jobs координируют работу, а бизнес-правила живут в отдельных модулях; по возможности side effects собраны в sink-like компоненты, а не размазаны по длинным sync chains.
- Чем помогает: агенту проще делать локальные изменения без захвата инфраструктурного шума и без дорогого impact analysis по всей цепочке.
- Чем опасно отсутствие: FastAPI route, Celery task или CLI-команда одновременно валидирует, считает бизнес-логику, пишет в БД и шлёт интеграции; изменение в начале pipe затрагивает полсистемы.
- Сложность внедрения: средняя, а в legacy может быть высокой.
- Как проверять: читать handlers и смотреть, являются ли они thin layer; отдельно смотреть, не превращён ли каждый сервис в звено длинной цепочки синхронных вызовов с побочными эффектами.
- Red flags: route handler > 100 строк; одна функция одновременно вызывает БД, внешние API, форматирование ответа и бизнес-валидацию.

## 10. Тесты рядом с кодом и рядом с контрактом

- Что проверяем: тесты легко найти по месту, они документируют сценарии модуля и подтверждают инварианты.
- Чем помогает: агент быстрее находит способ проверить изменение и понимает intended behavior.
- Чем опасно отсутствие: verify-loop дороже, а знание о поведении модуля уходит в головы команды.
- Сложность внедрения: низкая или средняя.
- Как проверять: взять модуль и посмотреть, виден ли ему ближайший тест или надо идти в отдельный лес из папки `tests/` и угадывать соответствие.

## 11. Единый стиль упаковки и импортов для Python

- Что проверяем: проект использует предсказуемую схему пакетов, импортов и конфигурации инструментов, желательно без нескольких конкурирующих стандартов.
- Чем помогает: агент меньше тратит контекст на понимание инфраструктурной рутины.
- Чем опасно отсутствие: один модуль импортируется как пакет, второй как относительный путь, третий запускается через особый PYTHONPATH-трюк.
- Сложность внедрения: средняя; часто начинается с выравнивания `pyproject.toml`, `pytest`, `ruff`, `mypy` и package layout.
- Как проверять: смотреть, есть ли единый layout и один понятный способ запуска тестов, приложения и фоновых задач.

## 12. Метрики структуры как сигнал, а не украшение

- Что проверяем: команда отслеживает хотя бы часть структурных метрик, если архитектура уже сложная: size thresholds, cyclomatic complexity, coupling, instability, abstractness, package drift, distance from main sequence.
- Чем помогает: проблемы видны до того, как превратятся в хроническую хрупкость кода; AI slop заметен не только субъективно, но и численно.
- Чем опасно отсутствие: архитектурный долг замечают только по субъективному ощущению или после большого сбоя; brute-force AI-generated code остаётся незамеченным.
- Сложность внедрения: средняя.
- Как проверять: смотреть, есть ли threshold-based checks. Практические ориентиры: `CC < 10` для обычной функции — хороший базовый порог, `CC < 5` — предпочтительно, `CC > 50` — аварийный сигнал; доля функций с `CC > 10` должна быть известна и не расти. Для зрелых систем полезно отслеживать abstractness/instability и долю компонентов, дрейфующих в Zone of Pain.
- Red flags: рост доли функций с высоким CC после массовых AI-правок; низкоабстрактный «цементный» пакет, от которого зависят почти все остальные.

## Базовые выводы для аудита

- если architecture map не совпадает с путями в коде, сначала чинят structural alignment;
- если модуль нельзя понять без чтения десяти соседних файлов, проблема не только в docs, но и в устройстве кода;
- если архитектурное правило не проверяется автоматически, оно со временем будет нарушено;
- если verify зависит от глубокого tribal knowledge, AI-friendly кодовая база ещё не сложилась.
